# إعادة تصميم صفحة التوثيق في نور — متوافقة مع إضافة v7

## الهدف

تحويل صفحة التوثيق (`NoorPage.tsx`) من "لوحة تحكم ترسل أوامر للإضافة عبر جسر DOM" إلى "لوحة مراقبة تعرض الحالة الحية وتتيح للمستخدم التحكم في السجلات قبل التوثيق". الإضافة v7 تعمل مستقلة عبر API مباشر — لا تحتاج اتصال مع صفحة التطبيق.

## النطاق

### داخل النطاق
- إعادة هيكلة `NoorPage.tsx` (حذف كود الجسر + إضافة ميزات جديدة)
- إضافة تبويب "الموثق اليوم"
- إضافة إجراءات "استبعاد" و "حذف من التوثيق" للسجلات
- قسم "المستبعد" أسفل كل تبويب
- تحديث فوري عبر SignalR
- نقاط API جديدة على السيرفر (exclude, restore, documented-today)
- تحديث تصميم إضافة نور (ui.css, popup.css) لمطابقة لغة التصميم

### خارج النطاق
- تعديل منطق الإضافة v7 (noor-xhr.js, noor-content.js, etc.)
- إضافة ميزات استخراج جديدة
- تعديل صفحات أخرى في التطبيق

### ملاحظة: مسار ملفات الإضافة
ملفات الإضافة v7 في: `d:/وكيل شؤون الطلاب/اضافات الكروم/اضافة التطبيق شؤو ن الطلاب/noor-extension-v7/`

## 6. قسم "المستبعد"

يظهر أسفل جدول كل تبويب (1-4). لا يظهر في تبويب "الموثق اليوم".

### السلوك:
- **يظهر فقط** إذا فيه سجلات مستبعدة في هذا التبويب
- **قابل للطي/التوسيع** — افتراضياً مطوي
- العنوان: `block` "المستبعد (3)" مع سهم `expand_more` / `expand_less`
- الخلفية: `#fffbeb` (أصفر خفيف) لتمييزه عن الجدول الرئيسي
- `border-radius: 12px`، حدود `1px solid #fde68a`

### محتوى القسم:
جدول مبسط بنفس أعمدة التبويب الأصلي لكن بدون خانات اختيار. كل سجل فيه:
- زر **"إرجاع"** (أيقونة `undo`) — يعيد السجل للقائمة الرئيسية (حالته ترجع "معلق")
- زر **"حذف من التوثيق"** (أيقونة `delete_forever`) — يختفي نهائياً

### تأثير الاستبعاد على الإضافة:
الإضافة تجلب السجلات من `GET /api/noor/pending-records` الذي يستثني المستبعد. لذلك السجلات المستبعدة **لا تظهر للإضافة تلقائياً** بدون أي تعديل في كود الإضافة.

## 7. التحديث الفوري عبر SignalR

### المبدأ
لا تحديث دوري. التحديث يصير بطريقتين فقط:

### أ) فوري عبر SignalR

```
الإضافة توثق مجموعة في نور
    ↓
الإضافة ترسل: POST /api/noor/update-status
    ↓
السيرفر يحدّث قاعدة البيانات
    ↓
السيرفر يرسل إشعار SignalR:
  { type: "noor-status-updated", data: { updated: N, failed: N } }
    ↓
NoorPage.tsx تستقبل الإشعار عبر useSignalR()
    ↓
تعيد جلب السجلات والإحصائيات فوراً
    ↓
السجلات الموثقة تختفي من التبويبات 1-4 وتظهر في "الموثق اليوم"
```

### ب) يدوي
زر "تحديث" في شريط الإجراءات يجلب أحدث البيانات فوراً.

### مؤشر آخر تحديث
نص صغير بجوار زر التحديث يعرض: "آخر تحديث: قبل X ثوانٍ" — يتحدث كل ثانية.

### التعديل المطلوب في السيرفر

يُحقن `NotificationService` (الموجود فعلاً في المشروع) في `NoorController` بدلاً من `IHubContext` مباشرة — للتوافق مع نمط الكود الحالي.

في `POST /api/noor/update-status` بعد تحديث قاعدة البيانات:
```csharp
await _notificationService.SendAsync("noor-status-updated", new {
    updated = updatedCount,
    failed = failedCount
});
```

### حماية من التعارض (Race Condition)

عند استلام `update-status` من الإضافة، يجب التحقق من الحالة الحالية قبل التحديث:
- إذا كان السجل حالته `"مستبعد"` أو `"لا يحتاج"` → **لا يُحدّث** — يُرجع في عدد `failed`.
- إذا كان حالته `""` أو `"معلق"` أو `"failed"` → يُحدّث بشكل عادي.

هذا يمنع الحالة: المستخدم استبعد سجل في التطبيق بينما الإضافة كانت توثقه.

```csharp
// قبل التحديث:
if (record.NoorStatus == "مستبعد" || record.NoorStatus == "لا يحتاج")
{
    failedCount++;
    continue; // لا تحدّث
}
```

### التعديل المطلوب في NoorPage.tsx
```typescript
const { lastNotification } = useSignalR();

useEffect(() => {
  if (lastNotification?.type === 'noor-status-updated') {
    loadRecords(activeTab);
    loadStats();
  }
}, [lastNotification]);
```

## 9. تعديلات السيرفر

### حالات NoorStatus

| الحالة | المعنى | يظهر في الإضافة | يظهر في صفحة التوثيق |
|--------|--------|-----------------|---------------------|
| `""` أو `"معلق"` | جاهز للتوثيق | نعم | نعم — التبويبات 1-4 |
| `"مستبعد"` | مستبعد مؤقتاً | لا | نعم — قسم المستبعد |
| `"لا يحتاج"` | لا يحتاج توثيق نهائياً | لا | لا |
| `"تم"` | تم التوثيق بنجاح | لا | نعم — تبويب الموثق اليوم |
| `"failed"` | فشل التوثيق | نعم (يُعاد) | نعم — تبويب الموثق اليوم |

### نقاط API جديدة

#### `POST /api/noor/exclude`
استبعاد سجلات — حالتها تصبح "مستبعد".

طلب:
```json
{
  "updates": [
    { "id": 123, "type": "violation" },
    { "id": 456, "type": "absence" }
  ]
}
```

رد:
```json
{ "success": true, "data": { "updated": 2 } }
```

#### `POST /api/noor/restore`
إرجاع سجلات مستبعدة — حالتها ترجع معلق ("").

طلب:
```json
{
  "updates": [
    { "id": 123, "type": "violation" }
  ]
}
```

رد:
```json
{ "success": true, "data": { "updated": 1 } }
```

#### `GET /api/noor/documented-today`
السجلات الموثقة اليوم.

معاملات: `?type=all` (اختياري — all, violations, compensation, excellent, absence)

رد: نفس شكل `pending-records` لكن يرجع سجلات حالتها "تم" أو "failed" من اليوم. يُضاف حقل `result` لكل سجل ("نجح" أو "فشل").

**تحديد "اليوم":** يعتمد على `RecordedAt >= today` (نفس منطق `pending-records` الحالي مع فلتر `filterMode=today`). السجلات القديمة الموثقة اليوم لن تظهر — وهذا مقبول لأن الفلتر "كل غير الموثق" يغطي الباقي.

#### ملاحظة: السجلات الفاشلة
السجلات بحالة `"failed"` تظهر في **مكانين**:
1. تبويب "الموثق اليوم" — للعرض مع علامة ✗
2. التبويبات 1-4 — لأن `pending-records` يرجعها (بحيث الإضافة تعيد محاولتها تلقائياً)

لا يحتاج المستخدم تدخل يدوي — الإضافة ستعيد المحاولة في التشغيل التالي.

### تعديل النقاط الحالية

#### `GET /api/noor/pending-records`
تعديل الفلتر: يستثني السجلات التي حالتها `"مستبعد"` أو `"لا يحتاج"`.

الفلتر الحالي:
```csharp
.Where(v => v.NoorStatus == null || v.NoorStatus == "" || v.NoorStatus == "معلق")
```

يصبح (يبقى كما هو — "مستبعد" و "لا يحتاج" مستثناة تلقائياً لأنها ليست null أو "" أو "معلق"). لكن نضيف `"failed"` للسجلات الفاشلة حتى تُعاد محاولتها:
```csharp
.Where(v => v.NoorStatus == null || v.NoorStatus == "" || v.NoorStatus == "معلق" || v.NoorStatus == "failed")
```

**مهم:** هذا التعديل يُطبّق على **جميع** استعلامات الكيانات الأربعة في `pending-records`:
1. `Violations`
2. `TardinessRecords`
3. `PositiveBehaviors` (تعويضية + متمايز)
4. `DailyAbsences`

وكذلك استعلامات الإحصائيات في `GET /api/noor/stats`.

#### `POST /api/noor/update-status`
بعد تحديث قاعدة البيانات، يرسل إشعار SignalR:
```csharp
await _hubContext.Clients.All.SendAsync("ReceiveNotification", new {
    type = "noor-status-updated",
    data = new { updated = updatedCount, failed = failedCount },
    timestamp = DateTime.UtcNow.ToString("o")
});
```

يقبل أيضاً حالة `"لا يحتاج"` كقيمة صالحة للحقل `status`.

## 10. تدفق العمل الكامل

```
① المستخدم يفتح صفحة التوثيق في التطبيق
    ↓
② يشوف كل السجلات المعلقة مجمّعة بالصف
    ↓
③ يستعرض القائمة ويتحكم:
   — سجل فيه مشكلة ← يستبعده (ينزل لقسم المستبعد أسفل الصفحة)
   — سجل ما يحتاج توثيق ← يحذفه من التوثيق (يختفي نهائياً)
   — سجل غير مطابق ← يشوف التحذير الأحمر ويقرر (يستبعد أو يحذف)
   — يقدر يرجع سجل مستبعد من قسم المستبعد بضغطة زر
    ↓
④ يفتح نظام نور في تبويب ثاني
    ↓
⑤ الإضافة تجلب السجلات المعلقة فقط (بدون المستبعد) وتوثقها
    ↓
⑥ الإضافة تخلص مجموعة ← ترسل النتيجة للسيرفر ← SignalR ← الصفحة تتحدث فوراً
    ↓
⑦ المستخدم يشوف السجلات تختفي من المعلق وتظهر في "الموثق اليوم"
    ↓
⑧ الفاشل يظهر في "الموثق اليوم" بعلامة ✗ ويبقى في pending-records للإعادة
```

## 14. مربع تأكيد "حذف من التوثيق"

عند الضغط على "حذف المحدد من التوثيق" (إجراء جماعي) أو زر الحذف لسجل واحد، يظهر مربع تأكيد:

- العنوان: "حذف من التوثيق نهائياً"
- النص: "سيتم إزالة X سجل من صفحة التوثيق نهائياً. السجلات ستبقى في النظام (صفحة المخالفات/الغياب) لكن لن تظهر هنا مرة أخرى."
- زرّان: "إلغاء" (ثانوي) + "حذف نهائياً" (أحمر)
- تصميم المربع: نفس تصميم `confirmOpen` الحالي في NoorPage.tsx (overlay blur + بطاقة بيضاء radius 20px)

ملاحظة: حالة "لا يحتاج" **غير قابلة للإرجاع من الواجهة**. لو أراد المستخدم إرجاع سجل محذوف يحتاج تعديل يدوي في قاعدة البيانات. لذلك مربع التأكيد ضروري.
